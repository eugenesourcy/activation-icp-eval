Context: WhatsApp direct is the strongest channel for quality SRs (4.6% CVR vs 0.5% for forms). But ~76% of WA leads ghost after 1-2 messages — often because the first response doesn't give them enough value to continue. The bot addresses this with category expertise upfront and conversational SR collection that doesn't feel like a form.

I. What We Did

We studied the gold standard conversation, Eugene's requirements document and key findings on lead behavior, and 13 raw WhatsApp conversations from the test period — plus the A/B experiment data (variants 1-6, CVR breakdown, quality SR classification). From these we built a conversational agent and iterated the prompt through 7 versions based on Karl's feedback. The current version (v7) was tested against 8 lead personas drawn from real WA chat data: 8/8 pass, avg score 8.8/10. Key changes in v7: prices-first (when lead names category, first response includes price range), 2-3 line message cap, one-liner answers for catalog/process questions. The iteration methodology is fully reproducible — test personas, prompt versions, and scoring rubric are documented in the GitHub repo.

III. Iteration: v6 → v7 (Based on Karl's Feedback)

Karl's feedback on the prototype: "not business logical." We interpreted this as: each turn carries dropout risk. If a lead reads our message and it doesn't either hook them (category expertise, specific prices) or qualify them (budget math, quantity check), we've wasted the turn and they may never reply again.

What Changed

v5 (Original Prototype)

  • 4-5 line messages, sometimes 6+
  • Asked for specs before giving prices
  • "What's the process?" got a multi-step explanation
  • "Catalog?" got a paragraph about why we don't have one
  • Narrowing questions before value delivery

v7 (Current)

  • Hard cap: 2-3 lines per message
  • Prices FIRST when category is named
  • "Process?" → one-liner redirect
  • "Catalog?" → "No fixed catalog — we source from 2,000+ factories. What do you need?"
  • Every turn: wow > qualify > advance SR

Scored Results (v7)

We built a rubric scoring each bot turn on 5 dimensions (0-2 each, max 10/turn): message length, value delivery, qualification, conversation discipline, and the "last message test" — if this is the last message the lead reads, did we hook them or learn if they're worth chasing?

III. Honest Limitations

The bot doesn't depend on Michael's knowledge The bot's conversational rules are principle-based — general sourcing expertise from the gold standard, not Michael-specific supplier relationships, past quotes, or client history. Michael's real conversations were used as test data (we replayed lead messages through the bot), not as training data for the bot's knowledge. A completely new lead the bot has never seen would get the same quality of response. We verified this with a separate test set of 6 new lead scenarios — performance was consistent.
We need more conversation data The bot was tested against ~13 real conversations from the test period. Eugene has agreed to share the remaining ~22 WhatsApp conversations from the live experiment. More data means more edge cases covered and a stronger bot. The testing process is designed for this — new conversations get turned into test personas and run against the prompt.
Price accuracy — LLM estimates vs reality The bot's price ranges come from the LLM's general knowledge, not Sourcy's supplier database. We spot-checked against live Alibaba listings: for stainless steel water bottles, the bot quoted $4-8/unit, while Alibaba shows $0.89-4.80 — right ballpark but missed the low end. For most product categories the ranges are directionally correct, but misquotes are possible. Real supplier data (via the planned Sourcy data skill) would eliminate this risk. In the meantime, leads should be told these are estimate ranges — final quotes come from the sourcing team.
Other open items
  • Simulated conversations only — real leads behave unpredictably. Needs live testing with real traffic.
  • No image handling tested — many leads send photos. CLI testing can't verify this.
  • Production resilience — concurrent sessions, API rate limits, and downtime handling need to be validated. Will need support from the tech team on production hardening.

IV. How to Keep Improving

The bot's conversational quality comes from two workspace files (SOUL.md + AGENTS.md) — plain text, editable, no code changes needed. Improvement follows a simple loop:

  1. Find an issue in production (e.g., "bot asked for budget when lead said they don't know")
  2. Write a test persona that reproduces it
  3. Fix the prompt (edit AGENTS.md)
  4. Re-run all personas to check nothing else broke

This process found and fixed 7 issues over 5 iterations. Each fix is a one-line change in the workspace file — no code deploys needed.

Ownership I'll own the prompt updates and testing process. To keep the bot improving, I'll need ongoing access to the full conversation data — both the remaining ~22 test-period conversations Eugene agreed to share, and new conversations as they come in from production. The more real data the bot is tested against, the fewer surprises in production.

WhatsApp ban risk Baileys is an unofficial WhatsApp Web API — Meta does not officially support bot usage on personal WhatsApp numbers. Eugene's team previously got a number banned within 1 week using Botpress. Mitigation for testing: use a dedicated number (not the main Sourcy number), keep message volume low. For production: the official WhatsApp Business API removes this risk entirely but has its own setup process. Decision for the team — see Step 3 below.
OpenClaw platform risks OpenClaw is a fast-moving open-source project (188K stars, 9,000+ commits). That's a strength (active development, many channels supported) but carries risks we should be aware of:
  • It's infrastructure, not a chatbot builder. OpenClaw behaves as a long-lived stateful backend service. Failures that don't show up in testing can surface under production load — message loss during gateway restarts, session isolation issues, and cron/heartbeat timing problems have been reported.
  • WhatsApp channel stability. Known issues: messages can be silently dropped during gateway restarts (config changes trigger restarts), and cron-triggered messages sometimes don't reach WhatsApp even when the system reports success.
  • Breaking changes. The project moves fast — updates can change config format, CLI behavior, or plugin APIs. We pin to a specific version and test before updating.
  • Single maintainer risk. Core development is driven by a small team. If the project stalls or pivots, we'd need to maintain our fork or migrate.
Mitigations: Pin to a tested version (currently 2026.2.10). Monitor the gateway with health checks. Keep the prompt/workspace portable — if we ever need to switch platforms, the conversational logic (SOUL.md + AGENTS.md) transfers to any LLM system. The investment is in the conversation design, not the framework.
